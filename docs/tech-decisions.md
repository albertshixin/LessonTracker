## 1. 非功能需求 (How Well)
* **及时性**：提醒推送必须精准，误差不能超过 1 分钟。
* **易用性**：打卡流程需在 3 次点击内完成（点击课程 -> 拍照 -> 确认）。
* **存储性能**：图片上传需进行本地压缩，避免消耗过多流量，加载历史记录时需流畅。
* **数据安全**：用户的上课照片（尤其是涉及未成年人）需安全存储，不作为公开数据。

## 2. 技术与约束 (Constraints)
* **推荐技术栈**：
    * **Flutter** 或 **React Native**：为了获得更好的原生硬件权限（相机、精准的本地通知 Alarm Manager），混合开发优于纯 Web。
    * 如果是微信小程序：需注意微信对“长期订阅消息”的限制，可能需要结合日历写入接口。
* **运行环境**：iOS 和 Android 手机。
* **数据存储**：
    * 结构化数据（课程信息）：本地 SQLite 或 云端数据库。
    * 非结构化数据（图片）：对象存储（OSS/S3）。

## 3. 可交付目标 (Done Definition)
第一阶段 MVP 验收通过的标准：
1.  **建档**：能成功创建一门“钢琴课”，总课时 20，规则为“每周六下午 3 点”，类型为“艺术类”。
2.  **提醒**：在周六下午 2:00（提前1小时）和 2:50（提前10分钟）能收到手机弹窗提醒。
3.  **打卡**：收到提醒后点开 App，拍摄一张钢琴照片并提交。
4.  **核算**：提交后，首页“钢琴课”剩余课时变为 19，且在历史记录中能看到刚才拍摄的照片。
5.  **规则验证**：若标记为“请假”，则剩余课时保持 20 不变。
